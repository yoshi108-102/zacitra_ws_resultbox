Zacitraの文献調査とか、Streamlitのアプリとかが混在していてだるいので、一括でみんなで管理できるようにアプリを作成する。

セキュリティを意識するためと、Terraformで一括管理したいのでAWS基盤を利用していく。

とりあえずCognitoを利用して認証基盤を作成、CloudFront + S3でHTMLを静的配信できるようにしておいた。

Terraformの勉強を兼ねているので、`main.tf`を見ていく。
```hcl
locals {
  common_tags = merge(
    {
      Project = var.project_name
      Managed = "terraform"
    },
    var.tags
  )

  site_origin_id      = "${var.project_name}-site-origin"
  cloudfront_base_url = "https://${aws_cloudfront_distribution.site.domain_name}"
}
```
これはTerraformのlocalブロックで、単純に変数を定義することが可能。
```hcl
resource "aws_s3_bucket" "site" {
  bucket = "${var.project_name}-site"

  tags = local.common_tags
}
```
ここで`local.common_tags`を使用している
TerraformではHCLという独自言語を用いて記述される。
`resource <リソースタイプ> <ローカル名>`
という形で記述されるのだが、https://developer.hashicorp.com/terraform/language/style
リソースタイプに変な名前を入れることは許されないらしい。
aws_s3_bucketを呼び出した時点でなにかしらのバケットが作られることが決まっていて。それをこのコード内ではsiteと呼称しますよという感じ。
```python
site_handle = create_resource(
  type = "aws_s3_bucket",
  bucket = "...",
  tags = {...}
)
```
疑似コードで書くとこんな感じになる。typeによってkwargsが異なっている。
ちょっと分かりづらいが、`aws_s3_bucket.site`というのはプロパティではなくリソースアドレスのことを指している。
```hcl
resource "aws_s3_bucket_public_access_block" "site" {
  bucket = aws_s3_bucket.site.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
```
これも`aws_s3_bucket_public_access_block`というリソースタイプに`site`というローカル名を参照させるようにしているという宣言。dependencyが明確にはされていないが、暗黙依存で十分

https://developer.hashicorp.com/terraform/language/block/resource
